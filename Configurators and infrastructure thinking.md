Configurators are rule-based design systems that generate solutions from predefined parameters, constraints, and component libraries. Their practical value is that they capture the tribal knowledge — the exact spacing, tolerances, and unwritten rules — that is impossible to document in a single unified place. The hybrid approach that's emerging is to combine rule-based precision for the deterministic parts with AI's high-level pattern recognition for context and style. Construction has more modular elements than it behaves as if it does: plant rooms, bridges, tunnel systems, MEP assemblies. The industry's habit is to custom-design each one from scratch.

The business model argument matters here. Patric Hellermann at Foundamental points out that software is typically 2% of a construction company's P&L, with materials at 50% and labour and machinery at 48%. Optimising only the 2% while calling it digital transformation is optimising the wrong problem. The shift toward Outcome-as-a-Service — where construction companies purchase better, more cost-effective results rather than software licenses — reflects this. If your technology can deliver a project better and faster, demonstrate it by actually delivering. Don't promise transformation; back it up with results.

**Infrastructure thinking is the design principle that connects these.** Toby Li at Shopify described the choice as: build NFT gating in two weeks, or spend two months building a platform layer from which anyone could build NFT gating in one hour. Build the infrastructure that makes the feature trivial. Once that layer exists, many more scenarios emerge without each one requiring a fresh build. The mental model: write code against an API that doesn't exist yet — define the correct client code first, then implement the infrastructure that makes it possible.

Applied to AEC, this means asking for every build decision: is this an experiment, a feature, or infrastructure? Features take advantage of existing infrastructure. Infrastructure enables a whole class of features. An internal marketplace of micro tools — small, purpose-built utilities created by engineers and computational designers closest to the workflow problems — is the practical version. Each successful tool leaves reusable infrastructure behind. Successful internal solutions can then evolve into external products.
